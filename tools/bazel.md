# `tools/bazel` wrapper — transparent Aspect routing

A drop-in `tools/bazel` shell script that lets developers keep typing `bazel` while still reaching Aspect-specific functionality (`lint`, `format`, `delivery`, …) and Aspect's value-added wrappers around `build` / `test` / `run`.

This is the **source of truth** for customers who want to adopt Aspect CLI features without forcing their teams to learn a new command name. Copy `tools/bazel` from this repo into your own workspace, adjust the verb lists at the top, commit it, and you're done.

## What it does

When Bazel (or Bazelisk) sees a file at `tools/bazel` in your workspace, it execs that script instead of the real bazel binary, passing the resolved bazel path as `$BAZEL_REAL`. This wrapper uses that hook to dispatch each command to the right tool:

| You type | What runs |
| --- | --- |
| `bazel lint` | `aspect lint` |
| `bazel format` | `aspect format` |
| `bazel delivery //...` | `aspect delivery //...` |
| `bazel build //... --keep_going --config=ci` | `aspect build //... --bazel-flag=--keep_going --bazel-flag=--config=ci` |
| `bazel test //... --test_output errors` | `aspect test //... --bazel-flag=--test_output=errors` |
| `bazel query 'deps(//foo)'` | `$BAZEL_REAL query 'deps(//foo)'` (vanilla bazel, unchanged) |
| `bazel info workspace` | `$BAZEL_REAL info workspace` |

The interesting case is `build` / `test` / `run`. The wrapper routes those through `aspect` so you pick up its DX improvements (artifact upload, GitHub PR comments, BES streaming, …), but **bazel-native flags keep working** — they're transparently rewritten as `--bazel-flag=<flag>` so aspect forwards them verbatim to Bazel.

## How flag rewriting works

The wrapper classifies every flag against two explicit lists embedded at the top of the script:

- `ASPECT_FLAGS` — every flag accepted by `aspect`, `aspect build`, `aspect test`, or `aspect run`. Generated by scraping `aspect <cmd> --help`.
- `BAZEL_VALUE_FLAGS` — every Bazel flag that takes a SPACE-separated value (so the wrapper knows when to consume the next argv token). Extracted from `bazel help build|test|run|startup_options --long` at **Bazel 9.1.0** — 371 flags.

The classification rules:

1. If a flag's name is in `ASPECT_FLAGS`, pass through to `aspect` unchanged. This covers global flags like `--task-key`, `--timing`, feature flags like `--artifact-upload:enabled`, etc. — both `--task-key=val` and `--task-key val` forms work.
2. Otherwise, treat as a Bazel flag and wrap. `--keep_going` → `--bazel-flag=--keep_going`. `--config=ci` → `--bazel-flag=--config=ci`. `--config ci` → `--bazel-flag=--config=ci` (next argv consumed and glued with `=`, because `--bazel-flag` is single-token only).
3. After `--`, everything passes through verbatim (positional targets, `run` arguments, etc.).
4. Flags **before the verb** get the same classification. Bazel flags get wrapped as `--bazel-startup-flag=...` and moved to the post-verb position where aspect expects them. Aspect global flags stay in front of the verb.

Unknown flags (typos, brand-new Bazel flags not yet in the embedded list) wrap as `--bazel-flag=<original>` and let Bazel produce the real error. Aspect CLI 1.x tried to be smart about unknown flags; it didn't go well.

To refresh the embedded lists when Bazel or aspect adds flags:

```sh
# Aspect flags
{
  aspect --help
  for v in build test run; do aspect $v --help; done
} 2>&1 | grep -oE '^\s+--[a-zA-Z][a-zA-Z0-9_:-]*' | sed -E 's/^[[:space:]]*--//' | sort -u

# Bazel value-taking flags
for h in build test run startup_options; do bazel help $h --long; done \
  | grep -oE '^  --[a-z_][a-z0-9_]*' | sed 's/^  --//' | sort -u
```

## Installing in your repo

Grab the script straight from this repo and drop it in your own `tools/` directory:

```sh
mkdir -p tools
curl -fsSL https://raw.githubusercontent.com/aspect-build/aspect-cli/main/tools/bazel -o tools/bazel
chmod +x tools/bazel
git add tools/bazel
```

Make sure you're using a Bazelisk (or recent Bazel) that respects `tools/bazel` — every Bazelisk release since 2019 does. No other changes needed.

## Trace output

When the wrapper routes a command **through aspect**, it prints a single grey trace line on stderr beforehand showing the resolved command. This makes the rewrite visible:

```
[tools/bazel] aspect build --bazel-flag=--keep_going --bazel-flag=--config=ci //...
→ 🎬 Running `build` task
…
```

When the wrapper forwards **straight to bazel** (e.g. `bazel info`, `bazel query`, anything under vanilla mode below), the trace is silent — that path is uninteresting and would just add noise.

Env vars:

- `ASPECT_WRAPPER_TRACE=1` — print the trace on every exec, including bazel forwarding. Also forces the line on even when stderr is not a TTY (useful for piping debug output to a file).
- `ASPECT_WRAPPER_QUIET=1` — suppress the trace entirely. Wins over `TRACE`.

The trace is silent under non-TTY stderr by default, so CI logs and command-substitution captures aren't polluted.

## Vanilla mode — keep `bazel build` literal

Some developers prefer their local `bazel build` / `bazel test` / `bazel run` to stay vanilla — they want a 1:1 Bazel experience for the common verbs and reach for `aspect build` directly when they want the wrapped behavior. Set:

```sh
export ASPECT_WRAPPER_VANILLA=1
```

…and the wrapper forwards `build`/`test`/`run` straight to `$BAZEL_REAL` with flags untouched. Aspect-only verbs (`lint`, `format`, `delivery`, anything in `ASPECT_VERBS`) still route through aspect — those have no bazel equivalent, and aspect tasks can be defined with arbitrary names in `.axl`, so the wrapper can't safely assume an unknown verb is something bazel would understand.

This is per-shell, per-developer. To make the choice repo-wide, edit `COMMON_VERBS` in the script (set to empty `()` to make every common verb route straight to bazel).

## How it avoids infinite recursion

When `tools/bazel` routes `build`/`test`/`run` through `aspect`, aspect then needs to spawn its own child `bazel` to do the actual work. If your `PATH` still has `tools/bazel` first (the normal case), that child `bazel` invocation re-enters the wrapper — which routes it back to `aspect` — which spawns `bazel` again — and so on forever.

The fix: aspect sets `ASPECT_CLI_RUNNING=1` on every child `bazel` it spawns. `tools/bazel` checks for this on entry and forwards straight to the real bazel (via `$BAZEL_REAL`) without going through any routing logic. The cycle is broken at the first hop. This matches the same pattern Bazelisk uses for `BAZELISK_SKIP_WRAPPER`.

**Customers should never set `ASPECT_CLI_RUNNING` manually** — it's an implementation detail of the aspect ↔ wrapper handshake. If you wrap `aspect-cli` in your own wrapper, propagate the variable through.

## Customizing

Two verb lists at the top of the script:

- `ASPECT_VERBS` — verbs that should forward to `aspect` verbatim (no flag rewriting). Defaults: `lint format delivery configure outputs init docs license buildifier gazelle`. Add or remove based on which Aspect features you want exposed. **These always route to aspect**, regardless of `ASPECT_WRAPPER_VANILLA`.
- `COMMON_VERBS` — verbs that route through `aspect` with bazel-flag rewriting. Defaults: `build test run`. Set to `()` to make every common verb stay vanilla regardless of env vars. Or leave the default and let individual developers opt out with `ASPECT_WRAPPER_VANILLA=1`.

Plus the two flag lists (`ASPECT_FLAGS`, `BAZEL_VALUE_FLAGS`) covered above.

## What it deliberately doesn't do

- **It doesn't fetch or pin Bazel.** That's Bazelisk's job. The wrapper just decides which tool to exec.
- **It doesn't second-guess unknown flags.** Anything not in `ASPECT_FLAGS` becomes `--bazel-flag=...` and Bazel produces the real error if it's a typo. Aspect CLI 1.x tried to be smart about unknown flags; it didn't go well.
- **It doesn't replace `bazel`.** You still install Bazel (via Bazelisk or otherwise). This script is purely an in-workspace dispatcher.

## Testing

Run `./tools/bazel-test.sh` (from the repo root) to exercise the wrapper against stubbed `aspect` and `bazel` binaries. There are 75 tests covering: verb dispatch (aspect-only, common, bazel-only), bazel flag wrapping (boolean, `=value`, space-value, repeated), aspect flag passthrough (kebab, colon-namespaced, boolean), pre-verb classification (startup flags, aspect globals, mixed), edge cases (`--`, hyphen-led targets, empty values, values with spaces), trace behavior, and `ASPECT_WRAPPER_VANILLA` mode. Add a `check` call when you change behavior.
